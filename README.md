# Code Assistant with Memory

**Код-ассистент с трехуровневой памятью и системой персонализации**

Интеллектуальный ассистент для разработки кода, использующий многоуровневую архитектуру памяти и поддерживающий несколько профилей пользователей с разными предпочтениями.

## Компоненты системы

### 1. Краткосрочная память (ShortTermMemory)

**Файл:** [`memory/short_term.py`](memory/short_term.py)

Хранит историю диалога с двумя уровнями:

- **Окно сообщений**: последние N сообщений (по умолчанию 5)
- **Суммаризация**: сжатое представление старых сообщений

```python
class ShortTermMemory:
    def add(role: str, content: str) -> None
    def get_recent_window() -> List[Dict]
    def get_context() -> str  # Возвращает суммаризацию + последние сообщения
    @property
    def summary() -> str
```

**Особенности:**
- Ленивая суммаризация (только когда нужно)
- Настраиваемый размер окна
- Возможность передать кастомную функцию суммаризации

---

### 2. Рабочая память (WorkingMemory)

**Файл:** [`memory/working.py`](memory/working.py)

Текущий контекст задачи. Использует конечный автомат (FSM) для управления состоянием.

**Состояния (TaskState):**
- `PLANNING` — анализ, декомпозиция, уточнение
- `EXECUTION` — написание кода, реализация
- `VALIDATION` — тестирование, проверка
- `DONE` — задача завершена

**Допустимые переходы:**
```
PLANNING → EXECUTION (когда план подтверждён)
EXECUTION → PLANNING (если архитектура неверна)
EXECUTION → VALIDATION (когда код написан)
VALIDATION → DONE (если тесты проходят и пользователь подтверждает)
VALIDATION → EXECUTION (если нужно исправить)
DONE → PLANNING (новая задача)
```

**Структура WorkingMemory:**
```python
@dataclass
class WorkingMemory:
    task: TaskContext           # Контекст задачи (FSM)
    files: List[str]            # Файлы в контексте
    blockers: List[str]         # Блокеры
    variables: Dict[str, Any]   # Произвольные переменные
```

---

### 3. Долговременная память (LongTermMemory)

**Файл:** [`memory/long_term.py`](memory/long_term.py)

Хранит важную информацию между сессиями. Использует SQLite.

**Типы записей:**
- `user_preference` — предпочтения пользователя
- `code_pattern` — переиспользуемые паттерны кода
- `arch_decision` — архитектурные решения
- `lesson_learned` — извлечённые уроки

**Методы:**
```python
def save(entry: MemoryEntry) -> int
def recall(query: str, entry_type: str = None, limit: int = 5) -> List[MemoryEntry]
def get_by_type(entry_type: str, limit: int) -> List[MemoryEntry]
def format_for_prompt(entries: List[MemoryEntry]) -> str
```

**Примечание:** В текущей версии используется простой текстовый поиск по ключевым словам. В будущем планируется добавление векторного поиска (RAG).

---

### 4. Менеджер профилей (ProfileManager)

**Файл:** [`memory/profile_manager.py`](memory/profile_manager.py)

Управляет персонализацией ассистента. Поддерживает:

- **Default профиль** — хранится в MD-файлах (`memory_profiles/default/`)
- **Пользовательские профили** — хранятся в SQLite (`agent_state.db`)

**Три компонента профиля:**
1. **style.md** — стиль общения, предпочтения по коду, стиль обучения
2. **constraint.md** — технические ограничения, стек, политики безопасности
3. **context.md** — контекст проекта, роли, цели

**Методы:**
```python
def list_profiles() -> List[Dict]
def get_active_profile() -> str
def set_active_profile(profile_id: str) -> bool
def create_profile(id, name, style, constraints, context) -> bool
def update_profile(profile_id, ...) -> bool
def delete_profile(profile_id) -> bool
def get_profile_for_prompt() -> str  # Возвращает объединённый prompt
```

---

### 5. Менеджер персистентности (PersistenceManager)

**Файл:** [`memory/persistence.py`](memory/persistence.py)

Управляет сохранением и загрузкой состояния между сессиями.

**Хранимые данные:**
- Сессии (`sessions`)
- История диалогов (`conversation_history`)
- Состояние рабочей памяти (`working_memory_state`)
- Суммаризации (`conversation_summary`)

**Все таблицы поддерживают поле `profile_id`** — это позволяет изолировать данные по профилям.

**Ключевые методы:**
```python
def create_session(title, profile_id) -> str
def list_sessions(profile_id, limit) -> List[Dict]
def save_message(role, content, session_id, profile_id)
def load_conversation(session_id, profile_id) -> List[Dict]
def save_working_memory(state_dict, session_id, profile_id)
def load_working_memory(session_id, profile_id) -> Optional[Dict]
def delete_session(session_id, profile_id)
def update_session_info(session_id, profile_id, ...)
```

### 6. Класс CodeAssistant

**Файл:** [`agent.py`](agent.py)

Главный класс, координирующий все компоненты.

**Инициализация:**
```python
assistant = CodeAssistant(session_id=None, profile_id=None)
```

**Основной метод:**
```python
def ask(user_input: str) -> str:
    """Обработка запроса пользователя"""
```

**Внутренняя логика:**
1. Восстановление релевантных воспоминаний из долговременной памяти
2. Построение system prompt (профиль + память + контекст задачи)
3. Построение user prompt (краткосрочная память)
4. Вызов LLM через OpenRouter
5. Обработка tool calls (обновление памяти, переходы состояний)
6. Сохранение состояния

**Инструменты (tools), доступные агенту:**

| Инструмент | Назначение |
|------------|------------|
| `update_working_memory` | Обновление контекста задачи |
| `save_to_long_term_memory` | Сохранение в долговременную память |
| `add_task` | Добавление подзадачи |
| `update_task_status` | Изменение статуса подзадачи |
| `add_blocker` | Добавление блокера |
| `resolve_blocker` | Разрешение блокера |
| `transition_state` | Переход между состояниями FSM |
| `update_current_step` | Обновление текущего шага |
| `set_expected_from_user` | Установка ожидания от пользователя |

## API и инструменты

### Работа с состояниями FSM

Агент автоматически управляет переходами между состояниями. Пользователь может:

1. **Явно указать переход** — сказать "начни выполнение", "переходи к тестированию"
2. **Неявно вызвать переход** — когда код написан, агент сам переходит в VALIDATION

**Примеры фраз для перехода в DONE:**
- "задача завершена"
- "всё готово"
- "принимаю"
- "done"
- "закрывай"

### Сохранение в долговременную память

Агент автоматически сохраняет важную информацию. Пользователь может явно попросить:

```
Запомни, что я предпочитаю async/await вместо callback'ов
```

Агент создаст запись типа `user_preference`.

### Управление подзадачами

Агент может декомпозировать задачу:

```
Разбей задачу на подзадачи: 
1. Настроить FastAPI
2. Создать модели БД
3. Реализовать CRUD
```

## Настройка профилей

### Default профиль

Хранится в [`memory_profiles/default/`](memory_profiles/default/) в виде Markdown файлов:

- **style.md** — коммуникация, код, обучение
- **constraint.md** — ограничения, стек, безопасность
- **context.md** — проект, роли, цели

Эти файлы автоматически копируются при создании нового профиля.

### Создание нового профиля

Через интерфейс:
1. Развернуть "✨ Create New Profile"
2. Ввести ID (латиница, без пробелов)
3. Ввести отображаемое имя
4. (Опционально) Редактировать поля перед созданием
5. Нажать "Create Profile"

Через код:
```python
profile_manager = ProfileManager()
profile_manager.create_profile(
    profile_id="my_profile",
    name="Мой профиль",
    style="...",
    constraints="...",
    context="..."
)
```

### Редактирование профиля

В интерфейсе:
1. Выбрать профиль в списке
2. Нажать "✏️ Edit Profile" (только для кастомных профилей)
3. Изменить поля
4. Сохранить или сбросить к default

## Примеры использования

### Пример 1: Написание кода

```
Пользователь: Напиши FastAPI приложение с CRUD для заметок
Ассистент: [Анализирует, переходит в PLANNING]
          Составляет план:
          1. Модель Note (id, title, content, created_at)
          2. Pydantic схемы (NoteCreate, NoteResponse)
          3. SQLAlchemy async engine
          4. CRUD операции
          5. API эндпоинты
          
          [Переходит в EXECUTION]
          Пишет код...
          
          [Переходит в VALIDATION]
          Проверяет, всё ли работает
          
          [Переходит в DONE]
```

### Пример 2: Работа с памятью

```
Пользователь: Я предпочитаю, чтобы все функции имели type hints
Ассистент: [Автоматически сохраняет в long-term memory]
          Запись типа user_preference создана

Пользователь: Напиши функцию для валидации email
Ассистент: [Восстанавливает из памяти предпочтение]
          Добавляет type hints ко всем параметрам и возвращаемому значению
```

### Пример 3: Использование нескольких профилей

```
Profile "Work":
- Стек: Python, FastAPI, PostgreSQL
- Стиль: краткий, по делу
- Ограничения: только open-source библиотеки

Profile "Learning":
- Стек: Python, базовый
- Стиль: подробные объяснения, примеры
- Ограничения: простые решения, без оптимизаций

Переключаемся между профилями → ассистент меняет поведение
```

## Ограничения и известные проблемы

1. **Векторный поиск** не реализован — используется простой LIKE-запрос
2. **Файловая система** — инструменты для чтения/записи файлов не добавлены
3. **Мультиагентность** — не реализована (в планах)
4. **Автоматические миграции** — только добавление колонок, удаление/изменение типов не поддерживается
